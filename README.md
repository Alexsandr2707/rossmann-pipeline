# Конвейер регрессии для Rossmann Store Sales

MVP MLOps-конвейера для табличной регрессии на наборе данных Kaggle Rossmann Store Sales.

## Установка

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

## Docker CI/CD

В текущей версии проект использует локальный CI/CD-процесс на основе Docker. `docker compose` описывает воспроизводимые сервисы для тестирования, обучения, потоковых обновлений, автономной оценки, построения прогнозов, генерации сводного отчета и короткой локальной CD-последовательности.

GitHub Actions в этот вариант намеренно не добавлен. Поэтому требования задания, которые явно требуют GitHub Actions, в текущей версии не закрываются и могут быть реализованы позже в отдельном рабочем процессе.

Docker-образ содержит код, конфигурацию и Python-зависимости. Исходные данные, обученные модели и результаты выполнения остаются вне образа: они зависят от окружения, могут занимать много места и должны сохраняться между запусками контейнеров. Compose подключает эти директории из локальной рабочей папки:

- `data/`
- `models/`
- `artifacts/`
- `reports/`
- `logs/`

Собрать локальный образ, если скачивание Docker-образов разрешено:

```bash
docker compose build
```

Запустить CI-сервис с модульными тестами:

```bash
docker compose run --rm ci
```

Запустить этапы конвейера через отдельные Compose-сервисы:

```bash
docker compose run --rm pretrain
docker compose run --rm train
docker compose run --rm update
docker compose run --rm evaluate
docker compose run --rm inference
docker compose run --rm summary
```

`pipeline` остается универсальной точкой входа для произвольных CLI-аргументов:

```bash
docker compose run --rm pipeline -mode inference -file data/external/test.csv
docker compose run --rm pipeline -mode update
docker compose run --rm pipeline -mode update 3
```

Несколько потоковых шагов обновления можно выполнить через `pipeline -mode update <count>` или повторным запуском сервиса `update`.

Локальный CD-сервис последовательно запускает тесты, предварительное обучение и генерацию сводного отчета в одном Compose-окружении:

```bash
docker compose run --rm local-cd
```

Работу локального Docker-процесса подтверждают следующие артефакты выполнения:

- `logs/pipeline.log`
- `models/*.pkl`
- `models/archive/*.pkl`
- `artifacts/collector_state.json`
- `artifacts/*history*.csv`
- `reports/summary_latest.md`

## Данные

Используется набор данных Kaggle Rossmann Store Sales: исторические ежедневные продажи 1 115 магазинов Rossmann. Задача проекта — прогнозировать значение колонки `Sales` для новых строк с учетом календарных признаков, факта открытия или закрытия магазина, промоакций, праздников, типа магазина, ассортимента и расстояния до ближайшего конкурента. В данных также встречаются временные закрытия магазинов, например из-за ремонта.

Ожидаемые исходные файлы:

- `data/external/train.csv`
- `data/external/store.csv`
- `data/external/test.csv` для построения прогнозов

Дополнительный справочный файл Kaggle:

- `data/external/sample_submission.csv`

Источник обучающих данных — `train.csv`. Перед предварительной обработкой он объединяется с `store.csv` по колонке `Store` через левое соединение многие-к-одному. Автономная оценка не использует `test.csv`.

Целевая переменная: `Sales`.

Временная колонка: `Date`.

Правило против утечки целевой информации: `Customers` никогда не используется как признак.

Кратко по файлам:

- `train.csv` — исторические данные с `Sales`;
- `test.csv` — данные без `Sales`, используемые для построения прогнозов;
- `store.csv` — дополнительная информация о магазинах;
- `sample_submission.csv` — пример Kaggle-отправки в нужном формате.

Ключевые поля:

- `Store` — идентификатор магазина;
- `Sales` — дневная выручка, которую прогнозирует модель;
- `Customers` — число покупателей за день; не используется как признак из-за риска утечки целевой информации;
- `Open`, `Promo`, `StateHoliday`, `SchoolHoliday` — признаки режима работы, промоакций и календарных событий;
- `StoreType`, `Assortment`, `CompetitionDistance` — характеристики магазина и конкурентного окружения;
- `Promo2`, `Promo2SinceYear`, `Promo2SinceWeek`, `PromoInterval` — признаки длительной повторяющейся промоакции.

## Архитектура проекта

Пакет `app/` разделен по зонам ответственности этапов конвейера. Структура хранит модули рядом с тем этапом, где они используются:

```text
app/
  core/           # CLI-оркестрация, чтение конфигурации и настройка логирования
  data/           # загрузка данных, сбор потоковых порций, проверка данных и признаки
  models/         # интерфейсы моделей, sklearn-реализации и блоки предварительной обработки
  training/       # обучение, обновление, диагностика и интерпретация
  serving/        # построение прогнозов
  evaluation/     # автономная оценка модели
  monitoring/     # метрики выполнения операций
  reporting/      # Markdown-отчеты и графики истории прогнозов
  visualization/  # переиспользуемые генераторы SVG-графиков
```

Логика группировки:

- `app/core/` содержит связующий код проекта: `config.py`, `logging_utils.py` и `pipeline.py`. Эти модули соединяют CLI с остальной системой, но не содержат ML-логики.
- `app/data/` содержит подготовку табличных данных до моделирования: загрузку CSV, объединение со сведениями о магазинах, разбиение периодов по датам, сбор потоковых порций, проверку качества данных, предварительную обработку исходных колонок и построение признаков.
- `app/models/` содержит переиспользуемые блоки для построения моделей: обертки, фабрики и совместимые со sklearn преобразователи предварительной обработки.
- `app/training/` отвечает за жизненный цикл модели: начальное обучение, потоковые обновления, диагностику и артефакты интерпретации модели.
- `app/serving/` отвечает за построение прогнозов. Он загружает обученную модель, применяет тот же путь построения признаков, что и при обучении, и записывает результаты прогнозирования.
- `app/evaluation/` отделен от обучения во время выполнения, потому что автономное сравнение моделей является аналитическим процессом, а не частью цикла потокового обновления.
- `app/monitoring/` записывает метаданные производительности на уровне операций, которые затем используются в отчетах.
- `app/reporting/` и `app/visualization/` отвечают за представление результатов: `reporting` собирает выходные данные, а `visualization` содержит переиспользуемый рендеринг графиков.

Внешняя точка входа остается небольшой. `run.py` разбирает CLI-аргументы, загружает конфигурацию из `app.core.config`, настраивает логирование и передает выполнение в `app.core.pipeline.Pipeline`.

## CLI

```bash
python run.py -mode update
python run.py -mode update 3
python run.py -mode pretrain
python run.py -mode evaluate
python run.py -mode inference -file data/external/test.csv
python run.py -mode summary
python run.py -mode summary -open
python run.py -mode reset
```

`inference` читает внешний CSV, например `data/external/test.csv`, объединяет его со `store.csv` по `Store`, если доступны сведения о магазинах, применяет тот же путь построения признаков, что и при обучении, загружает `models/best_model.pkl` и записывает исходные входные колонки вместе с `predict` в файл:

- `artifacts/predictions/inference_<timestamp>.csv`

`evaluate` - основной режим проверки качества. Он читает полный `train.csv`, объединяет его со `store.csv`, применяет то же разбиение периодов по датам, что и `pretrain`, обучает настроенные регрессионные модели на начальном периоде и сравнивает прогнозы на проверочном периоде с простыми базовыми моделями.

Результаты оценки:

- `reports/offline_model_evaluation.md`
- `reports/figures/offline_evaluation/actual_vs_prediction_timeline.svg`
- `artifacts/offline_model_evaluation.csv`

`summary` записывает сводный Markdown-отчет в `reports/summary_latest.md`. Команда `python run.py -mode summary -open` генерирует отчет и открывает его в приложении по умолчанию. Сводный отчет включает последние записи о выполнении операций (`inference`/`update`), историю качества данных, историю метрик модели, общий временной график прогнозов по всем запускам `update` и активные гиперпараметры модели из конфигурации.

Потенциально большие Markdown-таблицы в отчетах ограничиваются 5 строками: таблицы истории показывают последние 5 записей, а ранжированные таблицы показывают 5 лучших строк.

CLI-команды печатают основной результат в stdout:

- `update` печатает `True` или `False` для каждого выполненного обновления, затем печатает путь к обновленному Markdown-отчету;
- `inference` печатает путь к CSV-файлу с прогнозами;
- `summary` печатает путь к Markdown-отчету.

Команда `python run.py -mode update 3` обрабатывает до трех потоковых порций подряд. Выполнение останавливается раньше, если новых порций нет. После завершения каждого действия конвейер автоматически регенерирует сводный Markdown-отчет. Актуальный отчет находится в `reports/summary_latest.md`, а каждая генерация дополнительно сохраняется в `reports/archive/summary/`.

## Обучение

Настройки моделей находятся в `config/config.yaml`.

`pretrain` делит уникальные даты исходного набора данных на начальный, проверочный и потоковый периоды. Он обучает выбранную модель на начальном периоде, оценивает ее на проверочном периоде, затем заново обучает стартовую потоковую модель на данных начального и проверочного периодов. После успешного предварительного обучения сохраняется `current_model.pkl`, а состояние потока инициализируется так, чтобы следующий `update` начался с первой потоковой даты.

`pretrain` рассчитан на запуск после `reset`. Если в `models/` уже есть `.pkl`-файл модели, команда завершится ошибкой и предложит выполнить `reset`, чтобы случайно не перезаписать существующий жизненный цикл модели.

Активные регрессионные модели:

- `decision_tree_regression`
- `knn_regression`
- `ridge_regression`
- `random_forest_regression`

Автономная оценка обучает все настроенные `candidate_models`, поэтому сравнивает активный набор моделей и базовые модели даже тогда, когда обучение во время выполнения использует `training_mode: single`.

Конвейер использует предварительную обработку sklearn:

- числовые колонки: заполнение пропусков медианой и стандартизация;
- категориальные колонки: отдельная категория для пропусков и частотное кодирование;
- колонки, полученные из `Date`: год, месяц, квартал, день и номер недели ISO.

Режим `update` во время выполнения эмулирует поток данных, записывая следующие `model.stream_batch_days` уникальных дат из потокового периода:

- `data/raw/batch_XXXX.csv`
- `data/processed/batch_XXXX_processed.csv`
- `artifacts/collector_state.json`
- `artifacts/batch_metadata_history.csv`
- `artifacts/data_quality_history.csv`
- `artifacts/performance_history.csv`
- `artifacts/model_metrics_history.csv`
- `reports/archive/eda/eda_batch_XXXX.md`
- `reports/eda_latest.md`
- `models/archive/model_vXXXX_<model_name>.pkl`
- `models/best_model.pkl`
- `reports/summary_latest.md`
- `reports/archive/summary/summary_<YYYY-MM-DD_HH-MM-SS>_<operation>.md`
- `reports/model_diagnostics_latest.md`
- `reports/figures/model/prediction_timeline.svg`
- `models/current_model.pkl` после `pretrain` или `update`

Регрессионные метрики:

- `rmse`
- `mae`
- `r2`
- `smape`
- `pearson_corr`
- `pearson_p_value`

Основная метрика выбора модели: `rmse`.

Метрики качества данных включают долю пропусков, дублирующиеся строки, константные колонки, изменение схемы данных, долю выбросов по IQR и мощность категориальных признаков. Сводные отчеты читают эти метрики из `artifacts/data_quality_history.csv`.

Мониторинг производительности записывает метаданные о длительности и статусе операций в `artifacts/performance_history.csv`, а сводные отчеты показывают последние строки из этого файла.

`reset` удаляет сгенерированные артефакты выполнения и сохраняет исходные CSV-файлы, конфигурацию, код и файлы `.gitkeep`.
